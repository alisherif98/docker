Chapter 1

What is Docker?

Before we jump into commands, let's understand what Docker actually is and the problem it solves. Skip this if you already know — but if you're new, please read this carefully. Everything else will make so much more sense.

The problem Docker solves

Imagine you build an application on your laptop. It uses Python 3.10, PostgreSQL 14, some environment variables, and a specific version of a library like numpy 1.24. It works perfectly on your machine. 🎉

Now you send it to your colleague. On their laptop, they have Python 3.9, no PostgreSQL, different libraries. The app crashes. This is the famous "it works on my machine" problem.

Docker fixes this by packaging your application together with everything it needs — the OS libraries, the runtime, the dependencies, the config — into a single portable unit called a container. Anywhere Docker runs, your container runs the same way. Your laptop, your colleague's laptop, a server in the cloud, a Raspberry Pi. Same behavior everywhere.

💡 Simple analogy

Think of a container like a shipping container 📦. Whether you ship it by truck, train, or boat — the container is identical, standardized, and its contents don't change. Docker does the same for software: your app + its environment, packaged so it runs identically anywhere.

Container vs Virtual Machine

People often confuse containers with VMs. They're both ways to isolate software, but they're very different:

FeatureVirtual MachineDocker Container
Includes OSYes — full guest OS (Windows/Linux)No — shares the host kernel
SizeGigabytes (2–20 GB typical)Megabytes (5–500 MB typical)
Startup timeMinutesMilliseconds to seconds
Resource useHeavy (dedicated RAM/CPU)Light (shares host resources)
Isolation levelVery strong (hypervisor)Strong (Linux namespaces/cgroups)
Best forFull OS environments, mixed OSApps, microservices, dev environments
Architectural comparison: VMs virtualize hardware; containers virtualize the operating system.
Chapter 2

Why should you use Docker?

Docker isn't just a fad. It has real, tangible benefits — especially if you're a developer, data engineer, DevOps, or student learning modern tools. Here's why it matters:

  1. Consistency across environments. Your app runs identically on your laptop, in staging, and in production. No more "works on my machine".
  2. Fast onboarding. New team member? They run docker compose up and the whole stack (app + DB + cache + queue) starts working in seconds.
  3. Isolation. Run Python 3.8 for one project and Python 3.12 for another, side by side, with zero conflicts.
  4. Portability. Move your app from AWS to Azure to GCP without rewrites. Same container everywhere.
  5. Scalability. Need 10 copies of your app to handle traffic? Run 10 containers. Orchestrators like Kubernetes make this automatic.
  6. Reproducibility. The Dockerfile is a recipe. Anyone who runs it gets the exact same environment. Great for science, ETL pipelines, and audits.
  7. Efficiency. Containers use fewer resources than VMs — one server can run dozens of containers where it'd run only 2–3 VMs.
🎯 Real-world example

You're a data engineer building an ETL pipeline with Airflow, Postgres, and dbt. Instead of installing all three on your laptop (versions conflict, breaks other projects), you write a docker-compose.yml. Then docker compose up gives you all three running in seconds — and you can throw it all away with docker compose down when you're done. Clean.

Chapter 3

Core Concepts you must know

Docker has a handful of terms that get used constantly. Understanding these unlocks everything else. Don't skip.

🖼️ Image

An image is a read-only template or blueprint. It contains: the app code, the runtime (Python, Node, Java, etc.), the libraries, the OS files, and configuration. Think of it like a class in programming — a definition. Or like an installer file (.iso, .exe) but for containers.

Examples: nginx:1.25, python:3.12-slim, postgres:16-alpine. These are all images you can download.

📦 Container

A container is a running instance of an image. If image = class, container = object. If image = installer, container = installed & running program. You can have many containers running from the same image (like many instances of the same class).

🔑 Key insight

Image → Container is like Recipe → Cake. The recipe (image) tells you how to make it. The cake (container) is the actual thing you eat. From one recipe, you can bake many cakes.

🐳 Dockerfile

A Dockerfile is a plain text file with instructions on how to build a custom image. It says: "start from this base image, copy these files, install these packages, run this command when starting." We'll cover this in detail in Chapter 12.

🗂️ Registry

A registry is a storage & distribution system for Docker images. The most famous one is Docker Hub — like GitHub, but for images. You can docker pull images from it and docker push your own.

Other registries: GitHub Container Registry (ghcr.io), AWS ECR, Azure ACR, Google Artifact Registry, or your own private one.

💾 Volume

Containers are ephemeral — when you delete one, all its data is gone. If you want to persist data (a database, uploaded files), you use a volume: managed storage that lives outside the container's lifecycle.

🌐 Network

Containers can talk to each other over Docker networks. By default, containers on the same custom network can find each other by name. So a web app can connect to a database container by using db as the hostname.

🚢 Docker Compose

A tool for defining and running multi-container applications. Instead of typing many docker run commands, you describe your whole stack in a docker-compose.yml file and run docker compose up.

The complete Docker workflow at a glance.

Installing Docker

You'll want Docker installed to follow along. Below are the essentials for each OS. The commands assume you're on Linux (which is where Docker feels most native), but Mac and Windows work too.

1

Linux (Ubuntu / Debian / Fedora)

Use the official convenience script — the fastest way to get started:

bash
# Download & run the official install script
curl -fsSL https://get.docker.com | sh

# Add your user to the docker group so you don't need sudo
sudo usermod -aG docker $USER
newgrp docker

# Verify installation
docker --version
docker run hello-world
2

macOS & Windows

Download Docker Desktop from docker.com. It includes the engine, CLI, Compose, and a GUI. Just install and launch.

3

Verify everything works

Run this — if you see "Hello from Docker!", you're done. 🎉

bash
docker run hello-world
⚠️ Linux users only

If you get a "permission denied" error, you forgot the usermod step. Either add yourself to the docker group and log out/in, or prefix every command with sudo (not recommended long-term).

Chapter 5

Your first container — a hands-on tour

Let's actually run something. We'll use nginx (a web server) because it's tiny, useful, and shows off Docker's power in one command.

Run a web server in one command

bash
docker run -d --name mynginx -p 8080:80 nginx:alpine

Open your browser to http://localhost:8080. There's nginx! 🎉 Let's break down what happened:

PieceWhat it does
docker runCreate and start a new container
-dDetached mode — run in the background, so your terminal is free
--name mynginxGive the container a friendly name (otherwise Docker auto-generates one like zen_curie)
-p 8080:80Port mapping: connect port 8080 on your machine → port 80 inside the container
nginx:alpineThe image to use. nginx is the name, alpine is the tag (a lightweight variant)
💡 Why the port mapping?

Containers run in their own isolated network. Nginx listens on port 80 inside the container, but nothing outside can reach it unless you explicitly publish the port. -p 8080:80 tells Docker: "when someone hits port 8080 on the host, forward it to port 80 in the container".

Explore what's happening

bash
# List running containers
docker ps

# See the container's logs
docker logs mynginx

# Get a shell INSIDE the running container
docker exec -it mynginx sh

# Once inside, try:
# ls /usr/share/nginx/html    → nginx files
# exit                        → back to your host

Stop and clean up

bash
# Stop it (gracefully — sends SIGTERM, then SIGKILL after 10 seconds)
docker stop mynginx

# Remove the (now stopped) container
docker rm mynginx

# Or in one shot: stop + remove immediately
docker rm -f mynginx
🎓 What you just learned

You pulled an image from Docker Hub, ran it as a container, connected a network port, viewed logs, opened a shell inside it, and cleaned up. That's ~80% of daily Docker usage right there!

Chapter 6

Working with Images

Images are the blueprints for your containers. In this chapter you'll learn how to find them, pull them, list them, inspect them, tag them, and remove them.

Anatomy of an image name

An image reference looks like this: registry/repository:tag

  • Registry — where the image lives (default: Docker Hub, so often omitted). Example: ghcr.io, gcr.io.
  • Repository — the name (can include a namespace/user). Example: library/nginx, bitnami/postgresql.
  • Tag — the version. Default is latest. Example: 1.25-alpine, 3.12-slim.
examples
nginx                              # = docker.io/library/nginx:latest
nginx:1.25                         # specific version
python:3.12-slim                   # slim = smaller variant
ghcr.io/myuser/myapp:v2            # GitHub registry
myapp@sha256:abc123...             # immutable digest (pin exactly)
⚠️ Avoid :latest in production

latest is a moving target — today it might mean version 1.25, tomorrow 1.26. Always pin a specific tag (nginx:1.25.3-alpine) or, even better, a digest, so your deployments are reproducible.

Essential image commands

Download an image from a registry to your local machine.
FlagWhat it does
-aPull all tags of the repository
-qQuiet — suppress verbose progress output
--platformForce a specific architecture (e.g. linux/arm64, linux/amd64)
examples
docker pull nginx:alpine
docker pull python:3.12-slim
docker pull --platform linux/arm64 nginx:1.25-alpine  # for Apple Silicon / Raspberry Pi
docker images (alias: docker image ls)
List all images stored locally on your machine.
FlagWhat it does
-aShow all, including intermediate layers
-qOnly IDs (useful for piping into other commands)
--digestsShow content digests (the sha256 hash)
--filterFilter results: dangling=true, label=env=prod, reference=nginx*
--formatCustom output template (Go template syntax)
examples

docker image inspect
Get complete JSON metadata about an image — its environment variables, entrypoint, layers, labels, everything.
examples

docker image inspect -f '{{.Architecture}}' nginx:alpine  # just arch
docker tag
Give an image an additional name/tag. Doesn't copy — just an alias.
examples
# Tag your locally-built image for pushing to a registry
docker tag myapp:dev ghcr.io/myuser/myapp:v1.0.0
docker tag myapp:dev ghcr.io/myuser/myapp:latest
docker history
Show the layers of an image — every RUN, COPY, etc. that built it. Great for understanding image size.
examples
docker history nginx:alpine
docker history --no-trunc --format 'table {{.CreatedBy}}\t{{.Size}}' nginx
docker rmi (alias: docker image rm)
Remove one or more images from your local storage.
FlagWhat it does
-fForce removal even if a container is using it
--no-pruneDon't delete untagged parent images
examples

Save & load images (offline transfer)

Sometimes you need to move an image without internet access — e.g., to an air-gapped server. Docker has you covered:

bash
# On the source machine — export to a tar file
docker save -o nginx.tar nginx:alpine

# On the destination machine (after transferring the file)
docker load -i nginx.tar

# Save multiple images together
docker save -o backup.tar nginx:alpine redis:7 postgres:16
Chapter 7

Working with Containers

This is where you'll spend most of your time. Every container command orbits around docker run, but there are many nuances. Let's break it down carefully.

The lifecycle of a container

The container state machine. docker run = create + start.

The mighty docker run

This is Docker's most powerful command — and the one with the most flags. Rather than throw all 50+ at you, let's group them by purpose. Here's the anatomy:

syntax
docker run [OPTIONS] IMAGE [COMMAND] [ARG...]

🟢 Essential flags — you'll use these constantly

FlagWhat it doesExample
-dDetached — run in background-d
-itInteractive + terminal (for shells)-it ubuntu bash
--rmAuto-delete container when it stops--rm
--nameGive it a memorable name--name web
-p HOST:CTRPublish a port to the host-p 8080:80
-v SRC:DSTMount a volume or bind directory-v ./data:/data
-e KEY=VALSet an environment variable-e DB_HOST=db
--env-fileLoad env vars from a file--env-file .env

a realistic web app
docker run -d \
  --name webapp \
  -p 8080:3000 \
  -e NODE_ENV=production \
  -e DATABASE_URL=postgres://user:pass@db:5432/app \
  -v ./logs:/app/logs \
  --network appnet \
  --restart unless-stopped \
  myapp:1.0

🔒 Resource limits — protect your host

By default, a container can eat all your CPU and RAM. That's usually bad. Set limits:

FlagWhat it does
-m / --memoryHard memory limit (512m, 2g)
--memory-reservationSoft limit — kicks in only under pressure
--cpusMax CPUs, can be fractional (1.5)
--pids-limitMax number of processes/threads
--gpusGPU access: all, device=0
--ulimitStandard Linux ulimits (nofile=65536)
bash
docker run -d \
  --memory 512m \
  --cpus 1.5 \
  --pids-limit 200 \
  --ulimit nofile=65536:65536 \
  nginx:alpine

🔐 Security flags — always use these in production

FlagWhat it does
--user / -uRun as a specific UID/GID (never root!)
--read-onlyMake the root filesystem read-only
--cap-drop=ALLDrop all Linux capabilities
--cap-addAdd back only what you need (NET_BIND_SERVICE)
--security-opt no-new-privilegesPrevent privilege escalation
--tmpfs /tmpWritable in-memory tmp (works with read-only)
bash — hardened container
docker run -d \
  --user 1001:1001 \
  --read-only \
  --tmpfs /tmp:rw,nosuid,size=64m \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --security-opt no-new-privileges:true \
  nginx:alpine

♻️ Restart policies — auto-recovery

What happens if your container crashes? Or the machine reboots?

PolicyWhen to use
--restart noDefault. Don't restart.
--restart on-failureRestart only if exit code ≠ 0. Good for cron-like tasks.
--restart alwaysAlways restart, even after docker daemon restart
--restart unless-stoppedUsually what you want — like always but respects manual stops

Managing running containers

docker ps
List containers. Without flags, only running ones are shown.
FlagWhat it does
-aAll containers, including stopped
-qJust IDs
-sShow disk size
-lOnly the most recently created
--filterstatus=exited, name=web, label=env=prod
--formatCustom template
examples
docker ps                                        # running
docker ps -a                                     # all, incl. stopped
docker ps -a --filter status=exited              # only stopped
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
docker stop / start / restart / kill
Control the state of containers.
CommandBehavior
stopGracefully — SIGTERM, wait 10s, then SIGKILL
stop -t 30Give it 30 seconds to shut down cleanly
startStart a stopped container
restartStop + start
killImmediate SIGKILL — no grace period. Or send another signal with -s

examples

docker rm
Delete stopped containers permanently.
examples
docker rm web                          # must be stopped
docker rm -f web                       # force (works on running)
docker rm -fv $(docker ps -aq)         # nuke ALL containers + anon volumes
docker update
Change resource limits on a running container without recreating it.
examples
docker update --cpus 2 --memory 1g web
docker update --restart unless-stopped web
🚀 Handy shortcut

Use --rm for one-off tasks so you don't pile up stopped containers: docker run --rm -it python:3.12 python — gives you a Python REPL that vanishes when you exit.

Inspecting & Debugging Containers

Something's not working? These are the commands you'll use to figure out what's happening inside a container.

docker logs
Fetch stdout/stderr from a container. Your first stop when something's broken.
FlagWhat it does
-fFollow — stream new logs live (like tail -f)
--tail NLast N lines only (or all)
--since 10mOnly logs from the last 10 minutes
--untilStop at a timestamp
-tShow timestamps
examples
docker logs web                          # everything so far
docker logs -f web                       # follow live
docker logs -f --tail 100 web            # last 100, then follow
docker logs --since 10m -t web           # last 10min with timestamps
docker exec
Run a command inside a RUNNING container. Essential for debugging.
FlagWhat it does
-itInteractive with a TTY (for shells)
-uRun as a specific user
-wSet the working directory
-eSet an environment variable
-dDetached mode
examples
docker exec -it web bash                 # get a shell
docker exec -it web sh                   # alpine images: use sh, not bash
docker exec -u root web apt update       # run as root
docker exec web ls /var/log              # one-off command
docker exec -it db psql -U postgres      # psql shell in postgres container
docker inspect
Get low-level JSON info about ANY Docker object — container, image, network, volume.
examples
docker inspect web                                          # everything
docker inspect -f '{{.NetworkSettings.IPAddress}}' web      # just IP
docker inspect -f '{{.State.Status}}' web                   # running? exited?
docker inspect --type=network mynet                         # works on networks too
docker stats
Live CPU, memory, network, and disk I/O for containers. Like top for containers.
examples
docker stats                             # live, all containers
docker stats --no-stream                 # one snapshot then exit
docker stats web db                      # just specific containers
docker top / cp / diff
More debugging tools.
examples
docker top web                            # processes inside the container
docker cp web:/etc/nginx/nginx.conf ./    # copy file OUT of container
docker cp ./patch.sh web:/tmp/            # copy file INTO container
docker diff web                           # files changed since image
🕵️ Debugging workflow

When something breaks:
1. docker ps -a — is it running? What was the exit code?
2. docker logs -f — what did it print?
3. docker exec -it name sh — poke around inside.
4. docker inspect — check config, network, mounts.
5. docker stats — is it out of memory?

Chapter 9

Volumes & Data Persistence

Containers are ephemeral: when you delete one, all its data goes with it. That's usually fine — but not for databases, uploads, or logs. Volumes solve this.

Three ways to persist data

TypeWhat it isWhen to use
Named VolumeManaged by Docker, stored at /var/lib/docker/volumes/. Docker takes care of it.Default choice — databases, app data, anything you want to persist.
Bind MountA specific folder on your host, mounted into the container.Development (mount your code for live reload), config files.
tmpfs MountStored in RAM, gone when container stops.Sensitive data (secrets in memory), temporary caches.
Three mount types feed data into a container.

Working with named volumes

docker volume create / ls / inspect / rm
Manage the lifecycle of named volumes.
bash
docker volume create mydata            # create
docker volume ls                       # list all
docker volume inspect mydata           # details (where is it on disk?)
docker volume rm mydata                # delete
docker volume prune                    # delete all unused volumes

Using volumes in docker run

There are two syntaxes: the short -v shortcut and the verbose --mount. Both work, but --mount is clearer for production.

bash — the -v shortcut
# Named volume: mydata → /var/lib/postgresql/data inside container
docker run -d -v mydata:/var/lib/postgresql/data postgres:16

# Bind mount: current folder → /app inside container (dev workflow)
docker run -d -v $(pwd):/app node:20

# Read-only bind mount
docker run -d -v $(pwd)/config:/etc/nginx:ro nginx
bash — the verbose --mount syntax
# Named volume
docker run -d --mount type=volume,source=mydata,target=/var/lib/db postgres:16

# Bind mount (read-only)
docker run -d --mount type=bind,source=/host/code,target=/app,readonly node:20

# tmpfs — RAM only, 100 MB max, mode 1770
docker run -d --mount type=tmpfs,destination=/cache,tmpfs-size=100m myapp
✅ Rule of thumb

Use named volumes for anything you need to persist (databases, uploads). Use bind mounts only for development (editing code from your host) or read-only config. Use tmpfs for secrets or temp scratch space.

Real example: persistent Postgres

bash
# Create a volume for the database data
docker volume create pgdata

# Start Postgres, telling it to store data in the volume
docker run -d \
  --name mypg \
  -e POSTGRES_PASSWORD=secret \
  -v pgdata:/var/lib/postgresql/data \
  -p 5432:5432 \
  postgres:16

# Even if you delete the container, the data lives on:
docker rm -f mypg
docker run -d --name mypg2 -e POSTGRES_PASSWORD=secret -v pgdata:/var/lib/postgresql/data postgres:16
# mypg2 sees all the data from before!
Chapter 10

Networking

How do containers talk to each other? To the outside world? To your host? Docker gives you full control over this with virtual networks.

Types of Docker networks

DriverWhat it doesUse when
bridgeDefault. A private virtual network on the host. Containers can talk to each other.Most cases — apps on a single host.
hostSkip network isolation — container uses host's network directly.Max performance, or when you need raw host access.
overlayMulti-host networking across a Swarm cluster.Docker Swarm services.
noneNo networking at all.Security-critical batch jobs.
macvlan / ipvlanContainer gets its own MAC/IP on your LAN.Legacy apps that need to look like real hosts.

DNS magic — the killer feature

When you connect containers to the same custom network, they can find each other by name. No IPs, no config — just names.

bash
# Create a network
docker network create myapp-net

# Run Postgres on it
docker run -d --name db --network myapp-net \
  -e POSTGRES_PASSWORD=secret postgres:16

# Run your app on the same network. It can reach Postgres at "db":
docker run -d --name web --network myapp-net \
  -e DATABASE_URL=postgres://postgres:secret@db:5432/postgres \
  -p 8080:3000 myapp:1.0

# Notice: DATABASE_URL uses hostname "db" — Docker's DNS resolves it!
💡 Why this matters

The default bridge network does NOT provide DNS between containers. That's why you should always create a custom network for connected apps. It's one docker network create away.

Network commands

docker network create
Make a new network.
FlagWhat it does
--driverbridge (default), overlay, host, macvlan
--subnetSpecify the IP range: 172.20.0.0/16
--gatewayCustom gateway IP
--internalNo external internet access (isolated)
--attachableAllow later manual attachment (for overlay)
examples
docker network create appnet
docker network create --subnet 172.25.0.0/16 --gateway 172.25.0.1 mynet
docker network create --driver overlay --attachable swarmnet
docker network ls / inspect / rm
Manage networks.
bash
docker network ls                                # list all
docker network inspect myapp-net                 # details, including connected containers
docker network inspect -f '{{json .Containers}}' myapp-net
docker network rm myapp-net                      # delete
docker network prune                             # delete unused
docker network connect / disconnect
Attach/detach a running container to/from a network.
bash
docker network connect --alias api mynet web     # with a DNS alias
docker network disconnect mynet web              # detach

Port publishing — from outside the host

By default nothing outside the host can reach a container. Use -p to expose ports:

bash
-p 8080:80            # host 8080 → container 80 (both TCP)
-p 127.0.0.1:8080:80  # bind only to localhost — NOT accessible from LAN
-p 80:80/udp          # explicit UDP
-p 8080-8089:80-89    # port range
-P                    # publish ALL EXPOSEd ports to random host ports
Chapter 11

Dockerfile — Building Your Own Images

Everything up to now was using other people's images. Now you'll learn to make your own. A Dockerfile is a plain text recipe that Docker follows to build an image, step by step.

Your first Dockerfile

Let's containerize a simple Python web app. Create a folder with these files:

app.py
from flask import Flask
app = Flask(__name__)

@app.route('/')
def home():
    return 'Hello from Docker! 🐳'

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=3000)
requirements.txt
flask==3.0.0
Dockerfile
# 1. Start from a base image (small Python 3.12)
FROM python:3.12-slim

# 2. Set the working directory (created if it doesn't exist)
WORKDIR /app

# 3. Copy dependency file first (cache-friendly — see below)
COPY requirements.txt .

# 4. Install Python dependencies
RUN pip install --no-cache-dir -r requirements.txt

# 5. Copy the app code
COPY . .

# 6. Document that this container listens on port 3000
EXPOSE 3000

# 7. What runs when the container starts
CMD ["python", "app.py"]

Now build and run it:

bash
# Build the image (the . means "build from current directory")
docker build -t myflask:1.0 .

# Run it
docker run -d --name flask -p 8080:3000 myflask:1.0

# Visit http://localhost:8080 — Hello from Docker! 🐳

Dockerfile instructions — the complete reference

FROM
Base image to start from. Must be the first instruction (after optional parser directives).
examples
FROM python:3.12-slim
FROM node:20-alpine AS builder    # named stage (multi-stage)
FROM scratch                        # empty — you provide everything
FROM gcr.io/distroless/static       # minimal, no shell
WORKDIR
Set the current directory for subsequent instructions. Creates it if missing.
examples
WORKDIR /app
WORKDIR src        # relative → becomes /app/src
COPY vs ADD
Both copy files into the image. Use COPY by default. Use ADD only when you specifically need URL/tar features.
examples
COPY . /app                             # everything from build context to /app
COPY requirements.txt .                 # single file
COPY --chown=1001:1001 src/ /app/src/    # set ownership
COPY --from=builder /out/app /app       # from a previous stage
COPY --link ./static /static             # faster, deduped layer

ADD https://example.com/file.tar.gz /tmp/  # download from URL
ADD app.tar.gz /app/                       # auto-extract tar
RUN
Execute a command during the build. Each RUN creates a new image layer.
examples
RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*
RUN pip install --no-cache-dir -r requirements.txt

# BuildKit "heredoc" syntax — much cleaner for multi-line
RUN <<EOF
set -e
apt-get update
apt-get install -y curl nginx
rm -rf /var/lib/apt/lists/*
EOF

# Use build cache mount (BuildKit) — apt cache persisted across builds
RUN --mount=type=cache,target=/var/cache/apt apt-get install -y curl
💡 Layer minimization tip

Chain related commands with && in a single RUN to reduce layers. Each RUN adds a layer, and files deleted in a later layer still take up space in earlier ones!

ENV & ARG
Set variables. ARG is build-time only. ENV persists in the running container.
examples
ARG VERSION=1.0              # overridable at build with --build-arg
ARG TARGETPLATFORM             # predefined by BuildKit (multi-arch)

ENV APP_HOME=/app \
    NODE_ENV=production \
    PORT=8080
CMD vs ENTRYPOINT
Both tell the container what to run. Understand the difference — it trips up everyone.
InstructionPurpose
CMDDefault arguments. Fully overridable at docker run.
ENTRYPOINTThe main executable. Always runs. CMD becomes its arguments.
Example: CMD only
CMD ["python", "app.py"]

# docker run myimg              → python app.py
# docker run myimg sh           → sh (CMD replaced!)
Example: ENTRYPOINT + CMD
ENTRYPOINT ["python", "-m"]
CMD        ["http.server", "8080"]

# docker run myimg              → python -m http.server 8080
# docker run myimg module2 9000 → python -m module2 9000
⚠️ Always use exec form ([...])

Use the JSON array form CMD ["python", "app.py"] not shell form CMD python app.py. The shell form wraps in /bin/sh -c and breaks signal handling — your app won't receive SIGTERM properly on shutdown.

EXPOSE
Document which ports the container listens on. Doesn't actually publish them — just metadata.
examples
EXPOSE 8080
EXPOSE 443/tcp 53/udp
USER
Set the user for subsequent instructions and the running container. Never run as root in production!
examples
RUN groupadd -r app && useradd -r -g app -u 1001 app
USER 1001:1001          # prefer numeric IDs for Kubernetes compatibility
HEALTHCHECK
Tell Docker how to test if the container is actually healthy (not just running).
examples
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD curl -fsS http://localhost:8080/health || exit 1
VOLUME
Declare that a path should be a volume. Signals intent — data at this path should persist.
examples
VOLUME ["/data", "/var/log"]
LABEL
Add metadata to the image. Follow the OCI standard for maximum tooling support.
examples
LABEL org.opencontainers.image.source="https://github.com/me/myapp" \
      org.opencontainers.image.version="1.2.3" \
      org.opencontainers.image.licenses="MIT"

Multi-stage builds — small production images

Building an app often needs big toolchains (compilers, dev libraries). But the final image should be small. Multi-stage builds solve this: one stage builds, another packages the result.

Dockerfile — Go app example
# Stage 1: build (large image with Go compiler)
FROM golang:1.22 AS builder
WORKDIR /src
COPY . .
RUN go build -o /app/server ./cmd/server

# Stage 2: runtime (tiny image, just the binary)
FROM gcr.io/distroless/static
COPY --from=builder /app/server /server
USER 1001
EXPOSE 8080
ENTRYPOINT ["/server"]

Result: your build image might be 1 GB, but the final image is just ~10 MB. 100× smaller!

Best practices for Dockerfiles

✅ Do
  • Order layers from least-changing to most-changing (deps before code) — maximizes cache hits.
  • Use a .dockerignore.
  • Pin base image versions (python:3.12.4-slim).
  • Use multi-stage builds for smaller final images.
  • Run as non-root with USER.
  • Add a HEALTHCHECK.
  • Use exec form CMD ["...", "..."].
❌ Don't
  • Use :latest tags — they'll silently change under you.
  • Put secrets in ENV or files — they get baked into the layer forever.
  • Run as root unnecessarily.
  • Do apt-get update in a separate RUN from apt-get install — creates a stale cache.
  • Copy the whole context if you only need a few files.

BuildKit & Buildx — Modern Builds

BuildKit is Docker's modern build engine. It's faster, parallelizes stages, supports secrets and cache mounts, and can build multi-architecture images. Buildx is the CLI plugin on top of it.

Why BuildKit?

  • ⚡ Faster — parallelizes independent build stages.
  • 🔐 Secrets — inject secrets at build without baking them into layers.
  • 💾 Cache mounts — persistent apt/npm/pip caches across builds.
  • 🌍 Multi-arch — one command builds for amd64, arm64, arm/v7.
  • 📦 SBOM & provenance — supply-chain attestations built-in.

Enable BuildKit

bash
# On Docker Desktop, BuildKit is on by default.
# On Linux, enable it explicitly:
export DOCKER_BUILDKIT=1

# Or use buildx (recommended for new work)
docker buildx version

Create a multi-arch builder

bash
# Create a builder that supports multiple architectures (one-time setup)
docker buildx create --name mybuilder --driver docker-container --use --bootstrap

# Verify supported platforms
docker buildx inspect --bootstrap

Build for multiple architectures

bash
docker buildx build \
  --platform linux/amd64,linux/arm64,linux/arm/v7 \
  -t ghcr.io/me/app:1.0 \
  --push .

# --push is required for multi-arch (result goes straight to a registry)
# Use --load instead of --push to load a SINGLE-arch build into local docker

Registry cache — dramatically faster CI builds

bash
docker buildx build \
  --cache-from type=registry,ref=ghcr.io/me/app:cache \
  --cache-to   type=registry,ref=ghcr.io/me/app:cache,mode=max \
  -t ghcr.io/me/app:1.0 \
  --push .

Secrets during build (never leaked!)

Do NOT hardcode secrets in your Dockerfile. Use BuildKit's secret mount:

Dockerfile
# In your Dockerfile:
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm ci
bash
docker buildx build --secret id=npmrc,src=$HOME/.npmrc -t myapp .

The secret is mounted only during that RUN — never in a layer, never in image history. Perfect for private registry auth.

Cache mounts for package managers

Dockerfile
# Persist apt cache across builds
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
    apt-get update && apt-get install -y curl

# Persist pip cache
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

Docker Compose — Multi-container apps

Real apps aren't just one container — they're a web frontend + API + database + cache + queue. Docker Compose lets you define the whole stack in one YAML file and run it with a single command.

A complete example

Let's define a Flask app + Postgres + Redis:

docker-compose.yml
services:
  web:
    build: .
    ports:
      - "8080:3000"
    environment:
      DATABASE_URL: postgres://postgres:secret@db:5432/app
      REDIS_URL: redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: app
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5

  cache:
    image: redis:7-alpine
    volumes:
      - redisdata:/data

volumes:
  pgdata:
  redisdata:

Start everything with one command:

bash
docker compose up -d
# All three containers start. web waits for db to be healthy before starting.
# They can all reach each other by service name: "db", "cache", "web"

Compose commands you'll actually use

CommandWhat it does
docker compose up -dStart everything in the background
docker compose up --buildRebuild images first, then start
docker compose downStop & remove containers (keeps volumes)
docker compose down -vAlso delete named volumes — data loss!
docker compose psList services in the current project
docker compose logs -f webFollow logs of the web service
docker compose exec web bashShell inside a running service container
docker compose run --rm web pytestSpawn a NEW throwaway container to run a command
docker compose restart webRestart just one service
docker compose build --no-cacheRebuild ignoring cache
docker compose pullUpdate all images from registry
docker compose configValidate & render the effective YAML
docker compose watchLive-sync files & rebuild on change (dev)

Key concepts

🔗 depends_on with health conditions

Just declaring depends_on: [db] only waits for the db container to START, not to be ready. Use condition: service_healthy with a healthcheck for real dependency management.

🎨 Profiles — optional services

docker-compose.yml
services:
  debug-tools:
    image: alpine
    profiles: [debug]           # only starts if you ask for it
bash
docker compose --profile debug up    # now debug-tools is included

🔥 Compose Watch — live-reload for development

docker-compose.yml
services:
  web:
    build: .
    develop:
      watch:
        - action: sync         # copy files into running container
          path: ./src
          target: /app/src
        - action: rebuild      # rebuild image on dep change
          path: requirements.txt
bash
docker compose watch      # edits sync in real-time!
💡 Pro tip: layered override files

Keep a base docker-compose.yml and layer environment-specific overrides:

docker compose -f base.yml -f prod.yml up -d

Fields in later files override earlier ones. Great for dev/staging/prod separation.

Docker Swarm — Built-in Orchestration

You have a container running. What if you want 5 copies, spread across 3 machines, with automatic restart if they crash, and rolling updates without downtime? That's orchestration. Docker Swarm is Docker's built-in orchestrator. It's simpler than Kubernetes but capable enough for many teams.

🤔 Swarm vs Kubernetes?

Kubernetes is the industry standard for large-scale orchestration. Swarm is dead-simple, uses the same Compose format, and works great for smaller setups (up to a few dozen nodes). If you're just starting, Swarm is easier. Most production goes to Kubernetes eventually.

Set up a Swarm cluster

bash
# On the first machine — become the manager
docker swarm init --advertise-addr 10.0.0.5

# Get the token to add worker nodes
docker swarm join-token worker

# On each worker machine, run the printed command:
docker swarm join --token SWMTKN-1-... 10.0.0.5:2377

# Back on the manager, see all nodes
docker node ls

Deploy a service

bash
docker service create \
  --name web \
  --replicas 5 \
  --publish published=80,target=8080 \
  --env NODE_ENV=production \
  --limit-cpu 1.0 --limit-memory 512m \
  --update-parallelism 1 --update-delay 10s \
  --restart-condition any \
  myrepo/api:1.0

# List services
docker service ls

# See where the tasks are running
docker service ps web

# Scale up
docker service scale web=10

# Rolling update to a new version
docker service update --image myrepo/api:1.1 web

# Rollback if something went wrong
docker service rollback web

Deploy a full stack

bash
# Same docker-compose.yml you already have!
docker stack deploy -c docker-compose.yml myapp

# View / manage
docker stack ls
docker stack services myapp
docker stack ps myapp
docker stack rm myapp

Configure log rotation (essential!)

By default, json-file logs grow without bound. On a busy container this fills your disk. Always set rotation:

bash — per container
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  --log-opt compress=true \
  nginx
/etc/docker/daemon.json — global default
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3",
    "compress": "true"
  }
}

Chapter 18

Security Best Practices

Docker containers are strongly isolated, but they're not perfect. Follow these practices to avoid the most common pitfalls.

The essential checklist

✅ Production security checklist
  • Pin images by digest (myimage@sha256:...), not :latest
  • Run as non-root — use USER in Dockerfile or --user at runtime
  • Read-only root filesystem — --read-only plus --tmpfs /tmp
  • Drop all capabilities, add only what's needed — --cap-drop=ALL --cap-add=NET_BIND_SERVICE
  • Prevent privilege escalation — --security-opt no-new-privileges
  • Set resource limits — --memory, --cpus, --pids-limit
  • Add a healthcheck — HEALTHCHECK in Dockerfile
  • Rotate logs — max-size, max-file
  • Scan images in CI (Docker Scout / Trivy / Grype)
  • Sign images — Cosign / Docker Content Trust
  • Generate SBOM & provenance — supply chain proof
  • Never bake secrets into images — use BuildKit secrets or a vault
  • Don't mount /var/run/docker.sock unless you truly need to — it's root on the host

A fully-hardened container

bash
docker run -d \
  --user 1001:1001 \
  --read-only \
  --tmpfs /tmp:rw,nosuid,size=64m \
  --tmpfs /run \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --security-opt no-new-privileges:true \
  --security-opt apparmor=docker-default \
  --pids-limit 200 \
  --memory=256m --memory-swap=256m \
  --cpus=0.5 \
  --restart=unless-stopped \
  --health-cmd='curl -fsS localhost/health || exit 1' \
  nginx:alpine@sha256:abc123...

Scan for vulnerabilities with Docker Scout

bash
# Quick overview
docker scout quickview myimage:1.0

# Full CVE report (only critical & high)
docker scout cves myimage:1.0 --only-severity critical,high --only-fixed

# Compare two images
docker scout compare --to nginx:1.24 nginx:1.25

# In CI: fail the pipeline on critical findings
docker scout cves --exit-code --only-severity critical myimage:1.0

Sign images with Cosign (recommended)

bash
# Generate a keypair (one-time)
cosign generate-key-pair

# Sign an image after pushing
cosign sign --key cosign.key ghcr.io/me/myapp:1.0

# Verify
cosign verify --key cosign.pub ghcr.io/me/myapp:1.0

SBOM & provenance during build

bash
docker buildx build \
  --provenance mode=max \
  --sbom true \
  --push -t ghcr.io/me/myapp:1.0 .

Power User Tips & Best Practices

A collection of hard-won lessons from the trenches. Bookmark this.

🏷️ Image tagging & versioning
  • Never use :latest in production. Pin specific versions or digests.
  • Digest pinning: nginx@sha256:abc... — immutable, cryptographically verified.
  • Follow semver: myapp:1.2.3, myapp:1.2, myapp:1.
  • Use tags that convey intent: myapp:staging, myapp:prod.
⚡ Build speed
  • Order layers least-changing first. Copy requirements.txt before COPY . ..
  • Use .dockerignore — smaller context = faster upload to daemon.
  • Enable BuildKit (DOCKER_BUILDKIT=1) for parallel stages & cache mounts.
  • Use --cache-from/--cache-to in CI to reuse cache across builds.
  • Use COPY --link for immutable copies — much faster.
📉 Small images
  • Prefer alpine or slim variants of base images.
  • Use multi-stage builds — build in a big image, copy artifacts to a small one.
  • Use distroless or scratch for the final stage when possible.
  • Chain RUNs with && and clean up in the same layer (rm -rf /var/lib/apt/lists/*).
🔍 Debugging
  • Combine docker inspect with jq: docker inspect web | jq '.[0].NetworkSettings'
  • Use docker events to see everything happening in real time.
  • Use docker exec -it --user root when you need to install debug tools inside.
  • Try docker run --rm -it <image> sh to explore an image's contents.
🌐 Remote Docker & contexts
  • Manage remote Docker via SSH: docker context create prod --docker host=ssh://user@prod.example.com
  • Switch contexts: docker context use prod, then all commands hit the remote.
  • Or one-shot: docker --context prod ps.
🐛 Common gotchas
  • Alpine has sh, not bash. Use docker exec -it web sh or install bash.
  • Ctrl+C doesn't stop a detached container — you need docker stop.
  • ARG before FROM is global — re-declare it inside each stage to use it.
  • Bind mounts hide the container's files at that path — sometimes surprising.
  • Signals only reach PID 1 with exec form — use CMD ["python", "app.py"], not CMD python app.py.
  • Zombie processes? Add --init or use tini.
🔐 Security shortcuts
  • Base every production container on: --user 1001 --read-only --cap-drop=ALL --security-opt no-new-privileges.
  • Never use --privileged unless you're 100% sure why.
  • Scan images in CI: docker scout cves --exit-code --only-severity critical,high.
🎯 Development workflow
  • Use docker compose watch for live reload during development.
  • Use profiles to keep optional services out of the default startup.
  • Keep a base docker-compose.yml plus environment overrides (docker-compose.prod.yml).
  • For one-off tasks: docker compose run --rm web pytest — no leftover container.
